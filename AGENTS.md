## 项目定位

项目名称：AI 二次元 / OC 创作平台网站。

一句话定位：面向二次元用户、OC 创作者、同人作者、Coser、摊主、小型创作者团队、买家和创作者服务提供者的 AI 二次元 / OC 平台，帮助用户从角色灵感生成角色资产，并围绕角色资产继续完成创作、商品买卖、创作者接单和角色互动。

核心定位必须是：**以 AI 创作为第一入口的二次元 / OC 综合平台**。

项目长期目标包含完整平台能力：AI 创作、角色资产、商品买卖、创作者接单、角色互动、购物车、支付、订单、退款、履约、多币种和库存同步。

当前阶段仍然先完成网站前端开发与视觉设计，跑通可点击、可浏览、可演示的页面体验。后端、真实登录、数据库、支付、真实 AI 接口和供应链能力分阶段实现，不在第一阶段一次性完成。

## 核心用户

1. 普通二次元用户：快速生成自己的 OC、人设、头像方向和角色卡，用于社交平台展示或自娱创作。
2. OC / 同人创作者：长期保存角色设定、世界观、关系网、剧情片段和角色说话风格。
3. Coser / 摊主 / 社团：生成漫展摊宣文案、商品菜单、价格牌、预售文案和社交平台宣传内容。
4. 小型创作者团队：用 AI 降低创作、宣发、设计和周边转化成本。
5. 买家：浏览和购买数字商品、轻量实体周边与角色相关创作资产。
6. 创作者服务提供者：展示服务、作品集、档期和接单规则，承接头像、立绘、摊宣、周边设计等委托。

## 当前开发目标

现阶段优先完成一个高完成度的前端原型，覆盖创作、交易、接单、互动和我的五个主区域。完整后端产品按后续阶段推进。

成功标准：

- 页面结构完整，用户能从首页进入主要功能页。
- 所有核心页面都有真实可感知的布局、内容层级和交互状态。
- 表单、按钮、筛选、Tab、卡片、详情页等前端交互可以演示。
- 生成、保存、删除、复制等行为可以先用前端 mock 数据或本地状态模拟。
- 商品、服务、愿望单、购物车、订单、委托记录和互动会话可以先用前端 mock 数据或本地状态模拟。
- 页面在桌面端和移动端都可用。
- 视觉风格符合二次元 / OC 综合平台定位，不像通用 SaaS 模板，也不退化为单纯陪聊站或纯交易列表。

## VGen 设计参考

本项目的前端设计参考 `/Users/luoshenghui/Downloads/ppqq/vgen-design-code-analysis.md` 中对 VGen 的分析。学习其信息架构、组件组织、交易发现节奏和横向内容流，不复制 VGen 的 logo、品牌文案、图片素材、具体交易机制或具体业务。

可借鉴的方向：

- 首页即产品，不做冗长介绍型 landing page。
- 首屏要能直接进入创作、搜索、浏览示例内容。
- 用浅灰白背景承载高饱和内容资产，让角色卡、插画占位、周边预览和摊宣卡片成为视觉主角。
- 使用大圆角卡片、pill 搜索框、pill 按钮、圆形分类入口和横向滚动内容流。
- 桌面端保留完整顶部导航；移动端顶部简化，主导航可采用底部 tab bar。
- 首页多个 section 采用 discovery rail：横向浏览角色示例、创作模板、摊宣模板、周边方向。
- 分类和筛选使用 chips / tabs / orb row，先做前端本地筛选。
- 卡片需要包含可扫读信息：标题、标签、状态、简短说明、示例结果、收藏或保存入口。

不要照搬的内容：

- 不把首页做成纯 marketplace 交易首页；AI 创作和角色资产仍是第一入口。
- 第一阶段不做真实订单、购物车、创作者后台、支付、多币种、私信或通知系统；后续阶段可以按完整平台目标实现。
- 不复制 VGen 的品牌立场、文案、图片、图标和具体色值。
- 不把首页做成纯商品列表；本项目必须围绕“OC 创作、角色资产、商品买卖、创作者接单、角色互动”组织内容。

本项目首页应更接近“二次元创作与交易发现页”：用户一进入页面先看到创作入口、角色卡示例和模板分类，同时能继续探索商品、创作者服务和角色互动。

## 前端 MVP 主路径

首版前端优先保留并强化创作主链路：

```text
用户进入首页
→ 点击进入创作页
→ 输入角色灵感
→ 看到模拟生成的 OC 人设
→ 查看角色卡结果
→ 保存到前端角色库
→ 进入角色详情
→ 继续生成剧情 / 对话风格 / 头像提示词 / 摊宣文案 / 商品菜单
```

同时新增三条可演示平台链路：

```text
用户进入首页
→ 进入逛商品
→ 浏览数字商品 / 实体周边预览
→ 查看商品详情
→ 加入愿望单或前端购物车状态
```

```text
用户进入首页
→ 进入找创作者
→ 浏览创作者服务
→ 查看创作者主页
→ 发起前端演示委托请求
```

```text
用户进入首页
→ 进入角色互动
→ 选择自己的角色资产
→ 输入场景或关系提示
→ 查看模拟对话结果
```

这一阶段可以使用 mock 数据、静态示例、前端状态或 localStorage。不要为了第一阶段前端原型提前引入数据库、真实登录、支付或复杂 API。

## 页面结构

### `/`

目标：参考 VGen 的 discovery homepage 逻辑，让用户快速进入创作、搜索模板、浏览示例角色，并继续探索商品、创作者服务和角色互动。

包含：

- Sticky 顶部导航：Logo、创作、逛商品、找创作者、角色互动、我的
- Hero：强一句话定位 + pill 搜索框 + 立即创建按钮
- Hero cards：创作助手、角色资产、摊宣工具三张横向卡片
- 分类 orb row：OC、人设、头像方向、同人剧情、摊宣、周边预览
- 角色卡示例 rail：横向展示多个示例角色卡
- 创作模板 rail：角色卡模板、剧情模板、对话风格模板、头像提示词模板
- 摊宣 / 周边 rail：商品菜单、价格牌、小红书文案、微博文案、徽章、立牌、小卡
- 商品 rail：数字商品、模板包、头像套装、徽章、立牌、小卡、贴纸
- 创作者服务 rail：头像委托、角色立绘、摊宣设计、周边图案、文案代写
- 角色互动入口：从自己的角色资产开始互动
- 移动端底部导航：首页、创作、逛商品、找创作者、互动、我的
- 定价入口可放在我的或页脚

## 代码规范

### 基本原则

- 先完成可用页面，再做抽象。
- 只写当前页面和交互需要的代码，不做超前封装。
- 组件拆分以复用和可读性为准，不为单次使用创建复杂抽象。
- 页面文件负责布局和组合，复杂 UI 片段下沉到组件。
- mock 数据集中管理，不要在多个组件里散落硬编码。
- 类型定义与实际 UI 使用保持一致，不提前设计后端字段。
- 不引入未说明的大型依赖。
- 不做无关格式化、无关重构或无关依赖升级。

### React / Next.js

- 优先使用现有项目技术栈；如果项目尚未开始，默认使用 Next.js + TypeScript + Tailwind CSS。
- 能用 Server Component 展示的静态内容不要强行改成 Client Component。
- 需要状态、事件、表单、localStorage 或浏览器 API 时再使用 Client Component。
- 组件 props 保持明确、简短，避免传入大而全对象后在组件内部猜字段。
- 列表渲染必须使用稳定 key。
- 表单必须有清晰的初始值、提交状态和错误提示。
- loading、error、empty、success 状态必须可见。

### CSS / Tailwind

- 优先使用项目已有设计系统、组件库和 Tailwind 配置。
- 不写大段重复 class；重复明显时再抽组件或提取样式。
- 保持移动端优先，所有核心页面必须适配小屏。
- 避免文字溢出、按钮挤压、卡片高度跳动和内容遮挡。
- 交互控件要有 hover、focus、disabled 状态。
- 视觉层级优先通过间距、字号、字重、色彩对比实现，不依赖过多装饰。

## 视觉与交互方向

整体风格参考 VGen 的轻量 marketplace / discovery 体验，但必须转译为以 AI 创作为第一入口的二次元 / OC 综合平台，而不是通用交易平台或单纯陪聊网站。

推荐：

- 浅灰白页面背景，给高饱和角色卡和作品占位留出舞台
- 明亮、干净、有创作感
- 轻微 ACG 视觉元素，避免低质装饰
- 大圆角卡片、pill 搜索框、pill 按钮和圆形分类入口
- 横向滚动 rail 作为首页和移动端的核心浏览模型
- 高对比 CTA
- 角色卡预览突出，创作结果比平台装饰更抢眼
- 商品、创作者服务和互动入口要从角色资产与创作需求自然延伸
- 登录界面可以使用更强的沉浸式主视觉，但必须服务账户入口，不做普通营销 hero
- 登录界面主视觉应制作成 6-8 秒无缝循环动态背景视频，并提供静态 fallback
- 移动端优先
- 内容密度适中
- UI 克制，让角色卡、插画占位、模板卡片、商品封面、服务样例和创作结果成为视觉主角

避免：

- 过度粉色恋爱风
- 低俗陪聊广告风
- 过多闪烁动效
- 信息堆叠混乱
- 只展示聊天框，不展示创作工作流
- 复制 VGen 的品牌文案、图片素材或具体交易结构
- 让交易、接单或聊天抢走 AI 创作第一入口

交互要求：

- 生成失败时展示可理解的错误信息。

- 当前阶段保存结果可以使用前端状态或 localStorage。
- 角色卡应适合截图、分享和导出。
- 首页搜索框、分类 chips、tabs、横向 rail、收藏 / 保存按钮必须有可演示的前端状态。
- 商品、创作者服务、愿望单、前端购物车、委托请求和互动会话必须有可演示的前端状态。
- 移动端横向 rail 的卡片宽度应露出下一张，给用户明确滑动暗示。

角色卡至少展示：

- 角色名

- 外貌
- 性格

摊宣输出适配：

- 微博：短标题、正文、话题标签。
- QQ 群：直接、清晰，包含时间、地点和价格。
- B 站动态：适合创作者语气和展示说明。

## Agent 工作规则

### 开始前

- 先阅读本文件，再修改项目。
- 明确当前任务的成功标准。
- 若需求存在多种解释，先指出分歧；能用更小、更稳方案解决时，优先选更小方案。
- 当前阶段默认选择前端实现，不主动引入后端方案；除非任务明确进入后端阶段。

### 修改时

- 只改和任务直接相关的文件。
- 不做顺手重构、无关格式化或无关依赖升级。
- 匹配现有代码风格和目录组织。
- 不引入未说明的大型依赖。
- 不破坏已有页面结构和用户路径。
- 新增页面必须保证移动端布局。
- 新增模拟生成功能必须同步补齐 loading、error、empty、result 状态。

### 完成前

- 运行与改动范围匹配的检查或测试。
- 页面类任务必须做浏览器视觉检查。
- 至少验证核心路径没有被破坏。
- 明确说明完成了什么、验证了什么、未验证什么。

## 首版验收标准

首版前端完成时至少满足：

- 用户能从首页进入创作页。
- 用户能输入关键词并看到模拟 OC 人设结果。
- 用户能查看角色卡结果。
- 用户能把结果保存到前端角色库。
- 用户能在角色库看到已保存角色。
- 用户能进入角色详情继续生成剧情、对话风格或头像提示词。
- 用户能生成摊宣文案、商品菜单和价格牌的前端模拟结果。
- 用户能查看定价页。
- 核心页面移动端可用。
- 所有生成过程有 loading、error、empty、result 状态。
- 整体视觉风格符合二次元创作工具定位。
